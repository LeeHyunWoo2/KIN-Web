## 전하고 싶은 말  요약 📢

저는

- **3개월** 만에 기존에 배운 기술과는 **성향이 매우 다른, 완전히 새로운 스택만으로 혼자서**,
- **설계부터 성능, 보안, UX, 유지보수까지 고려하고**,
- 각 인프라가 어떻게 상호작용하고, **데이터가 어떻게 흘러가는지 전부 파악**하며,
- 누가 가르쳐주지 않아도, 어떤 환경이든 **빠르게 적응하고 결과를 만들어낼 수 있는**,
- 말과 글이 아니라 **결과물로 증명하는 사람**입니다.

나머지 문서를 보실 때 **설계 과정**과 **제 사고방식**을 유심히 봐주세요.

## 프로젝트의 본질 🎯

이 프로젝트는 단순한 노트 앱이 아닙니다.<br/>
더 중요한 목표는 **"나"를 표현하는 것**입니다.

개발자로서, 얼마나 빠르게 새로운 기술을 습득하고, 이를 실전에서 활용할 수 있는지를 증명하고 싶었습니다.

**그래서:**<br/>
✅ 프론트엔드, 백엔드, 데이터베이스, 배포까지 **바닥에서부터 모든 것을 혼자 설계하고 구현**했습니다.<br/>
✅ 단순한 기능 나열이 아니라, **보안, 성능, UX, 유지보수까지 고민하며 설계**했습니다. 덕분에 트레이드오프 해결 경험이 꽤 있습니다.<br/>
✅ 특정 기술이나 프로그래밍 언어에 국한되지 않고, **어떤 환경에서도 빠르게 적응할 수 있는 역량을 강조**하려 합니다.

### 진짜 개발자는 "프로그래밍"만 잘하면 안된다.

저는 단순히 기술 스택을 익히는 것이 아니라,<br/>
- "**이 문제를 해결하기 위해 어떤 기술이 적절한가?**"
- "**해당 사항이 어디까지 영향을 끼칠 수 있는가?**" <s>(이런식으로 만들면, 고치면 나중에 무슨 사고가 날 수 있는가?)</s>

를 끊임없이 고민하며 학습했습니다.<br/>

이 프로젝트의 과정에서<br/>

 **보안** - 사용자 데이터와 백엔드 서버를 보호하기 위한 인증, 쿠키 보안, CORS, 세션 관리<br/>
 **성능** - 데이터 압축, 캐싱, API 최적화, 서버 부하 최소화<br/>
 **UX** - 단순한 UI 디자인이 아닌, 백엔드 설계부터 API 응답 속도, 데이터 구조, 인증 방식까지 사용자 경험을 고려한 설계 <br/>
 **유지보수** - 확장 가능한 코드 구조, 명확한 설계 패턴<br/>

등을 고려하며 **단순한 기능 구현이 아니라, 좋은 품질의 앱을 만들기 위해** 끊임없이 고민했습니다.

라이브러리를 선택할때도 단순히 구글에 검색하면 나오는 "이게 좋다더라~" 라는 글을 보고, 별 생각 없이 결정하지 않고<br/>
여러 후보를 정한 후 Github 와 공식 웹페이지의 Document를 살펴보면서 **무엇이 나에게 맞는지** 직접 판단했습니다.<br/>
덕분에 API Docs 를 활용하는 방법에 좀 더 익숙해졌습니다. 이게 정말 중요하다고 생각해요.<br/>
